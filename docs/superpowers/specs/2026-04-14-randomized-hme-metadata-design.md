# Randomized Hide My Email Metadata Design

## Summary

The generator currently submits the same `label` and `note` for every reserved Hide My Email address:

- `label`: `rtuna's gen`
- `note`: `Generated by rtuna's iCloud email generator`

This creates an obvious repeated pattern and also carries the wrong project branding. The new design replaces fixed metadata with natural-looking randomized English metadata generated fresh for each reserved address.

## Goals

- Generate a new `label` for every reserved Hide My Email address.
- Generate a new `note` for every reserved Hide My Email address.
- Keep the metadata human-readable and natural-looking in English.
- Ensure neither field contains `rtuna` or `spacex-3`.
- Remove `rtuna` branding from project documentation that is still user-facing.

## Non-Goals

- No external API, online dictionary, or AI dependency for word generation.
- No persistence of generated labels/notes across runs.
- No UI control for custom metadata patterns in this change.
- No large-scale refactor of generation flow outside the metadata path.
- No changes to the core Hide My Email generation request flow.
- No changes to iCloud login or authentication behavior.
- No changes to session storage, saved-account persistence, or related serialization logic.

## Current Behavior

Runtime metadata is currently defined in `icloud/hidemyemail.py`:

- `HideMyEmail.__init__()` defaults `label` to `rtuna's gen`
- `HideMyEmail.reserve_email()` sends a fixed `note`

This means every reserve request reuses the same visible metadata.

## Proposed Design

### 1. Per-reservation metadata generation

Create a small internal metadata generator in `icloud/hidemyemail.py` that produces a fresh `label` and `note` inside `reserve_email()` immediately before the request payload is built.

This keeps the randomness scoped to the exact reservation request instead of binding it to object construction time.

### 2. Natural-language word generation

Use local curated word lists of common English words and sentence templates.

#### Label format

- Built from 2 or 3 normal English words.
- Lowercase words joined by spaces.
- Examples of intended style:
  - `quiet harbor`
  - `silver meadow`
  - `clear morning field`

Constraints:

- Alphabetic words only.
- No gibberish or random character strings.
- No project or person branding.
- Short enough to look plausible as a user-entered label.

#### Note format

Use short natural English sentence templates such as:

- `Reserved for personal email routing.`
- `Created for inbox organization.`
- `Saved for private forwarding use.`

Optional template variants may reference the generated label if the wording stays natural, but the default preference is a neutral standalone sentence.

Constraints:

- Human-readable sentence.
- No brand or author names.
- No repeated fixed wording for every generated address.

### 3. README cleanup

Update `README.md` to remove remaining `rtuna` branding references that no longer match the maintained project identity.

The README should describe the project without legacy attribution text that conflicts with the current repository ownership.

## File-Level Changes

### `icloud/hidemyemail.py`

- Add local word pools and note templates.
- Add helper(s) to generate randomized metadata.
- Change `reserve_email()` to generate `label` and `note` per call.
- Remove the fixed default branding-oriented metadata path.
- Keep `generate_email()` request behavior unchanged.

### `README.md`

- Remove or rewrite the remaining `rtuna` references.

## Explicit Scope Guardrails

This change must stay limited to metadata generation and documentation cleanup.

The implementation must not modify:

- the core HME address generation logic,
- iCloud login, 2FA, or authentication flow,
- session persistence, saved sessions, or account serialization behavior.

### `tests/test_hidemyemail_metadata.py`

Add regression tests to prove:

- Generated metadata is human-readable and non-empty.
- Generated metadata does not contain `rtuna` or `spacex-3`.
- Repeated metadata generation is not fully constant.
- Reserve payload uses per-call generated metadata instead of one fixed literal pair.

## Testing Strategy

Follow TDD.

1. Add tests for metadata helper output format.
2. Verify the new tests fail for the current implementation.
3. Implement the smallest metadata generator that passes.
4. Add a reservation payload test proving fresh metadata is used.
5. Run the targeted tests, then the relevant broader test suite.

## Risks and Mitigations

### Risk: metadata still looks too repetitive

Mitigation:
- Use enough adjectives/nouns/templates to provide reasonable variety.
- Keep the structure simple and readable rather than overly clever.

### Risk: random output becomes unnatural

Mitigation:
- Only use curated word lists.
- Keep grammar templates narrow and predictable.

### Risk: tests become flaky because of randomness

Mitigation:
- Test invariants and diversity over multiple samples.
- Patch randomness where necessary for deterministic payload assertions.

## Success Criteria

The change is successful when:

- A single run that creates multiple addresses produces different labels/notes across reservations.
- All generated labels and notes remain readable English.
- No generated metadata contains `rtuna` or `spacex-3`.
- No runtime code path still sends the old fixed strings.
- The README no longer contains outdated `rtuna` branding.
